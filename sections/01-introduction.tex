\section{Introduction}
% \section{引言}

AI agents are widely used for coding, DevOps, and enterprise workflows\cite{yang2024sweagent,wang2025openhands,debenedetti2024agentdojo,zhan2024injecagent,agentsight}.
% AI 智能体被广泛用于编码、DevOps 和企业工作流\cite{yang2024sweagent,wang2025openhands,debenedetti2024agentdojo,zhan2024injecagent,agentsight}。
Besides the LLM, agents require \emph{harnesses}, software layers around the model that improve agent performance while enforcing instructions and constraints for safety and compliance~\cite{fowler-harness}.
% 除了 LLM，智能体还需要 \emph{harness}，即包裹模型的软件层，它在执行指令和约束以确保安全和合规的同时提升智能体性能~\cite{fowler-harness}。


A major component of the harness is a \emph{policy engine}, which observes and enforces instructions and constraints over the agent's concrete actions.
% harness 的一个主要组件是 \emph{策略引擎}，它观察并对智能体的具体行为执行指令和约束。
Projects encode many such policies in instruction files (e.g., \texttt{CLAUDE.md}, \texttt{AGENTS.md})~\cite{chatlatanagulchai2025manifests,chatlatanagulchai2025agentreadmes,santos2025claudeconfig,lulla2026agentsmd}, so harnesses can provide them to the model directly.
% 项目在指令文件（例如 \texttt{CLAUDE.md}、\texttt{AGENTS.md}）中编码许多此类策略~\cite{chatlatanagulchai2025manifests,chatlatanagulchai2025agentreadmes,santos2025claudeconfig,lulla2026agentsmd}，因此 harness 可以直接将它们提供给模型。
Because LLMs comply with instructions probabilistically~\cite{liu2024lost,jiang2024followbench,qi2025agentif} and may violate them due to planning errors, prompt-injection drift, and tool or script side-effects~\cite{greshake2023indirectprompt,zhan2024injecagent}, harnesses also require deterministic policy engines to enforce compliance~\cite{rebedea2023nemo,wang2025agentspec,debenedetti2025camel}.
% 因为 LLM 概率性地遵循指令~\cite{liu2024lost,jiang2024followbench,qi2025agentif}，并可能由于规划错误、提示词注入漂移和工具或脚本副作用而违反它们~\cite{greshake2023indirectprompt,zhan2024injecagent}，harness 还需要确定性策略引擎来强制合规~\cite{rebedea2023nemo,wang2025agentspec,debenedetti2025camel}。

    \centering
    \includegraphics[width=\linewidth]{figures/actplane-illustration.pdf}
    \caption{\sys{} enables the agent closest to the task to write concrete policy DSLs according to the higher authorities' instructions. The DSL is then compiled by \sys{} and enforced inside the OS kernel.}
    \label{fig:actplane-illustration}
\end{figure}

However, existing policy engines leave a \emph{semantic gap}: \emph{policy intent} is expressed in underspecified natural language, but enforcement must act on concrete \emph{system actions} based on project or task context.
% 然而，现有策略引擎留下了一个 \emph{语义鸿沟}：\emph{策略意图}以欠规范的自然语言表达，但执行必须根据项目或任务上下文对具体的 \emph{系统行为}做出判断。
For example, as illustrated in Figure~\ref{fig:actplane-illustration}, ``run tests before committing'' requires knowing which test command to run, and ``the worker sub-agent should not delete data files'' needs the sub-agent to locate where the data directories are.
% ``the reviewer sub-agent only writes to its project directory'' needs the sub-agent's assigned scope.
% 例如，如图~\ref{fig:actplane-illustration}~所示，"提交前运行测试"需要知道运行哪个测试命令，"工作子智能体不应删除数据文件"需要子智能体定位数据目录所在位置。
Our empirical study of CLAUDE.md and AGENTS.md instruction files in 64~popular projects (\S\ref{sec:empirical}) shows that enforcement must handle context-dependent system actions, with 64\% of statements being policies, 83\% involving system actions, and 74\% depending on context that cannot be pre-defined statically.
% 我们对 64 个热门项目中 CLAUDE.md 和 AGENTS.md 指令文件的实证研究（\S\ref{sec:empirical}）表明执行层必须处理依赖上下文的系统操作，其中 64\% 的语句是策略，83\% 涉及系统行为，74\% 依赖无法静态预定义的上下文。

Existing approaches fall short.
% 现有方法存在不足。
Tool-call guardrails~\cite{xiang2025guardagent,wang2025agentspec,shi2025progent,costa2025fides,debenedetti2025camel} miss indirect system actions that bypass the tool layer, such as a \texttt{git~commit} inside a script the agent wrote earlier.
% 工具调用护栏~\cite{xiang2025guardagent,wang2025agentspec,shi2025progent,costa2025fides,debenedetti2025camel}会遗漏绕过工具层的间接系统操作，例如智能体之前编写的脚本中的 \texttt{git~commit}。
OS-level enforcement systems (e.g., sandboxes~\cite{seccomp-bpf,apparmor,landlock,ciliumtetragon,gvisor}) expect static pre-defined policy, control resource access instead of actions, and return opaque denials (e.g., \texttt{EPERM}) without explaining which policy was violated or how to comply.
% 操作系统级执行系统（例如沙箱~\cite{seccomp-bpf,apparmor,landlock,ciliumtetragon,gvisor}）需要静态预定义的策略，控制资源访问而非操作，并返回不透明的拒绝（例如 \texttt{EPERM}），而不解释违反了哪条策略或如何合规。

To bridge the gap, we argue that an agent-harness policy engine should let agents define policies and enforce them at the OS level, where all execution paths are visible including subprocesses and shell-outs that bypass tool-call interception.
% 为弥合这一鸿沟，我们主张智能体 harness 策略引擎应该让智能体定义策略，并在操作系统级别执行，该层级可见所有执行路径，包括绕过工具调用拦截的子进程和 shell 调用。
While safety constraints such as ``never expose API keys'' come from higher authority such as the user, platform operator, or parent agent, the context needed to resolve policies resides with the agent closest to the task, which already reads the repository, interprets the current task, and resolves abstract references such as ``run tests'' into concrete commands.
% 虽然安全约束如"永不泄露 API 密钥"来自更高权限如用户、平台运营者或父智能体，解析策略所需的上下文存在于最接近任务的智能体处，该智能体已经读取了仓库、解释了当前任务，并将抽象引用如"运行测试"解析为具体命令。
This makes agents the natural producer of concrete policy, also reflecting the fact that instruction files are increasingly maintained by agents~\cite{anthropic2025claudecode,galster2026configuring}.
% 这使得智能体成为具体策略的自然生产者，也反映了指令文件越来越多地由智能体维护的事实~\cite{anthropic2025claudecode,galster2026configuring}。

Agent-authored, OS-enforced policy has two design requirements.
% 智能体编写、操作系统执行的策略有两个设计要求。
First, policy expressions must be high-level enough for agents to generate reliably, yet concrete enough to compile to deterministic kernel checks.
% 首先，策略表达必须足够高级使智能体能够可靠生成，同时又足够具体以编译为确定性内核检查。
Since 81\% of projects contain policies that define event ordering or data flow, they must also track state across operations.
% 由于 81\% 的项目包含定义事件顺序或数据流的策略，它们还必须跨操作追踪状态。
Second, agents must not weaken safety constraints from higher authority or affect other agents' policies.
% 其次，智能体不能削弱来自更高权限的安全约束或影响其他智能体的策略。

\sys{} is a programmable OS-level policy enforcement system for AI agent harnesses.
% \sys{} 是一个可编程的操作系统级策略执行系统，用于 AI 智能体 harness。
\sys{} provides a simple DSL for agents to express policies as deterministic kernel checks, such as \texttt{kill exec "git" "commit" unless after exec "go" "test" exits 0} (Figure~\ref{fig:actplane-illustration}).
% \sys{} 提供一个简单的 DSL，让智能体将策略表达为确定性内核检查，例如 \texttt{kill exec "git" "commit" unless after exec "go" "test" exits 0}（图~\ref{fig:actplane-illustration}）。
To enforce policies with event ordering or data flow, \sys{} compiles DSL rules into an eBPF engine that uses information-flow control (IFC), attaching labels that mark which sources have influenced each object and propagating them across process, file, and network operations.
% 为了执行具有事件顺序或数据流的策略，\sys{} 将 DSL 规则编译到使用信息流控制（IFC）的 eBPF 引擎中，附加标记哪些源影响了每个对象的标签，并跨进程、文件和网络操作传播它们。
Policies that express constraints use \texttt{block} or \texttt{kill} to deny violations with semantic feedback, while policies that express instructions use \texttt{notify} to guide the agent, e.g., ``blocked: commit without tests; run npm test first.''
% 表达约束的策略使用 \texttt{block} 或 \texttt{kill} 拒绝违规并提供语义反馈，而表达指令的策略使用 \texttt{notify} 指导智能体，例如"已阻止：未经测试的提交；请先运行 npm test。"
\sys{} uses policy domains bound to process subtrees to isolate agents from each other and prevent them from weakening higher-authority constraints.
% \sys{} 使用绑定到进程子树的策略域将智能体相互隔离，并防止它们削弱更高权限的约束。

On a decision-compliance benchmark, \sys{} resolves 2.0--3.2$\times$ more policy violations than prompt-filter, tool-regex, FIDES~\cite{costa2025fides} (tool-level IFC), and feedback-free kernel IFC by covering indirect execution paths that tool-call interception cannot observe, while adding 1.9\% end-to-end overhead on agent workloads and up to 8.4\% on kernel builds.
% 在决策合规基准上，\sys{} 通过覆盖工具调用拦截无法观察的间接执行路径，解决了比 prompt-filter、tool-regex、FIDES~\cite{costa2025fides}（工具级 IFC）和无反馈内核 IFC 多 2.0--3.2 倍的违规，同时在智能体工作负载上增加 1.9\% 的端到端开销，在内核构建上最高 8.4\%。
On a safety benchmark of 361 personal-assistant tasks, \sys{} prevents 74\% of baseline-unsafe behaviors by loading agent-generated safety policies as higher-authority rules before task execution.
% 在一个包含 361 个个人助理任务的安全基准上，\sys{} 通过在任务执行前将智能体生成的安全策略作为更高权限规则加载，阻止了 74\% 的基线不安全行为。
\sys{} is open-sourced at \url{https://github.com/eunomia-bpf/ActPlane}.
% \sys{} 已在 \url{https://github.com/eunomia-bpf/ActPlane} 开源。


To summarize, we make three contributions:
% 总结起来，我们做出三项贡献：
\begin{enumerate}
\item An \textbf{empirical study} of 64~projects that characterizes these gaps (\S\ref{sec:empirical}) and motivates \sys{}.
% \item 一项对 64 个项目的 \textbf{实证研究}，表征这些鸿沟（\S\ref{sec:empirical}）并激励 \sys{}。
\item \textbf{\sys{}}, a programmable OS-level policy enforcement system that addresses them (\S\ref{sec:design}--\ref{sec:implementation}).
% \item \textbf{\sys{}}，一个可编程的操作系统级策略执行系统，解决这些问题（\S\ref{sec:design}--\ref{sec:implementation}）。
\item An \textbf{evaluation} on a decision-compliance benchmark built on the empirical study, together with external coding-task and safety benchmarks covering workplace and personal-assistant tasks (\S\ref{sec:evaluation}).
% \item 一项在基于实证研究构建的决策合规基准上的 \textbf{评估}，连同涵盖工作场所和个人助理任务的外部编码任务和安全基准（\S\ref{sec:evaluation}）。
\end{enumerate}
