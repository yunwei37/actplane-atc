\section{Introduction}

% ¶1: Context — agents are powerful and widely deployed through harnesses
AI agents are widely used for coding,
DevOps, and enterprise workflows\cite{yang2024sweagent,wang2025openhands, 
debenedetti2024agentdojo,zhan2024injecagent,agentsight}, increasingly in production
deployments that run on real infrastructure, handle sensitive data, and
interact with CI/CD pipelines. Besides the LLM,
agents require AI agent harnesses: software around
the model that improves agent performance while enforcing behavioral
constraints for safety, security, and compliance~\cite{fowler-harness}.

% ¶2: Policy engines; instruction files are behavioral policy
A major component of such a harness is a policy engine~\cite{langchain-harness}: the
part that observes and enforces constraints over the agent's concrete
actions. In practice, projects encode many such constraints as
behavioral policies in instruction files
(\texttt{CLAUDE.md},
\texttt{AGENTS.md})~\cite{chatlatanagulchai2025manifests,lulla2026agentsmd}, so
harnesses can provide these instructions to the model directly.
Because LLMs comply with instructions
probabilistically~\cite{liu2024lost,jiang2024followbench,qi2025agentif}
and may violate them through planning errors,
prompt-injection drift, and opaque tool or script
behavior~\cite{greshake2023indirectprompt,zhan2024injecagent,debenedetti2024agentdojo},
harnesses also require deterministic policy engines to enforce compliance
at runtime~\cite{rebedea2023nemo,wang2025agentspec,debenedetti2025camel}.

% ¶4: Semantic gap — existing deterministic mechanisms also fail
Many of these policies constrain system-level actions, yet existing policy
 engines leave a \emph{semantic gap}:
\emph{AI agent intent and directives} are expressed in natural language,
 but enforcement must decide over
\emph{system actions}.
Our empirical study of instruction-files in 64~highly popular projects (\S\ref{sec:empirical}) 
finds that 64\% statements are behavioral directives, and 83\%
 involving system-level behavior: commands executed, files accessed,
and network connections made.
Existing approaches cannot close this gap.
Tool-call-based policy systems~\cite{xiang2025guardagent,wang2025agentspec,
shi2025progent,costa2025fides,debenedetti2025camel} in agent
runtimes such as Claude Code and Codex, whether LLM
guardrails or regex-like checkers, lose track of indirect system actions when
agents shell out or spawn subprocesses: the same \texttt{git~push} can
be reached through a \texttt{bash~-c} wrapper, or a
compiled binary created in a previous session; and at the repository
level, 81\% projects contain cross-event directives that single
tool-level interception cannot enforce (e.g., ``run tests before
commit'').

OS-level enforcement with static policy is also insufficient: 
most directives reference project or task context that 
static rules cannot resolve (e.g., ``never modify upstream source,'' 
where \emph{upstream source} must be resolved from the repository; 
``do not update dependencies unless requested''; 
``the reviewer sub-agent only writes to its project directory'').
We find that 74\% system-level directives
require context absent from low-level OS events.
OS-level enforcement systems and sandboxes for traditional
software~\cite{seccomp-bpf,apparmor,landlock,ciliumtetragon} can 
observe all system actions but expect
pre-defined static policy and return only system events without semantic
context, leaving the agent unable to follow its behavioral directives.

% ¶5: Insight — programmable policy engine below the tool layer
We argue that an agent-harness policy engine should be programmable by
agents and take effect at OS level: it should let agents
use live project and task context to enforce actions and adapt at
runtime. Our key observation is that, while safety constraints must 
come from higher authority, the required context is mainly within agents 
closest to the task: they already read
the repository, interpret the current task, and resolve abstract
references into concrete
commands and paths before enforcement can begin. This makes agents
the natural producer of concrete policy from directives, 
also reflecting the fact that instruction files are 
increasingly maintained by agents~\cite{anthropic2025claudecode,galster2026configuring}.

Realizing this creates three challenges. First, policy specification must be
high-level enough for agents to easily generate
from natural language with minimal token consumption, 
yet concrete enough to compile to
deterministic kernel checks. Second, 
enforcement must track
policy-relevant state across full system without imposing prohibitive per-syscall 
overhead. Third, since many rules are about safety and security, 
programmability must not become an authority bypass. Unlike 
traditional sandboxes threat model where a trusted
administrator authors policy, here agents are both author and
constrained subject, so they need an interface to add constraints
without weakening inherited policy set by trusted administrators or
parent agents or introducing safety risks.

% ¶6: System — \sys{}
\sys{} is a runtime-agnostic, programmable OS-level policy enforcement system for
AI agent harnesses. Agents express policies in a compact DSL 
generated from natural-language; \sys{} compiles them into labeled information-flow rules and
loads them into an eBPF enforcement engine across processes, files, and network kernel hooks.
When matching a rule, \sys{} returns semantic
feedback: which rule matched, why, and what to do next. Each rule 
independently selects one of three effects:
notify (observe), block (pre-operation denial), or kill (process 
termination). A layered authority model ensures that higher-authority
rules (e.g., global ``do not leak secrets'') cannot be weakened by
agent-authored policy.
On our decision-compliance benchmark,
\sys{} detects 2.4$\times$ more violations than prompt-filter or
tool-regex hooks by covering indirect execution paths that tool-call
interception cannot, while
adding 1.9\% end-to-end overhead on agent workloads and 8.4\% on
I/O-intensive kernel builds. On a third-party safety benchmark of 361
workplace and personal-assistant tasks, \sys{} prevents 76\% of
baseline-unsafe behaviors using automatically generated policies.

We make three contributions:
\begin{enumerate}
\item An \textbf{empirical study} of 64~projects that characterizes these
gaps (\S\ref{sec:empirical}).
\item \textbf{\sys{}}, a programmable OS-level policy enforcement system
that addresses them
(\S\ref{sec:design}--\ref{sec:implementation}).
\item An \textbf{evaluation} on our decision-compliance benchmark building on empirical study, external
coding-task and safety benchmarks covering
workplace and personal-assistant tasks.
(\S\ref{sec:evaluation}).
\end{enumerate}
