\section{Design}
\label{sec:design}

The empirical study establishes four requirements for 
a policy engine in AI agent harnesses:
\emph{agent-defined policy with semantic feedback}, 
since many rules need project or task context within the agent, 
and the agent must understand violations to recover;
\emph{OS-level cross-event enforcement}, since many
rules involve system-level behavior spanning multiple operations,
invisible to tool-call guardrails;
\emph{safety}, so enforcement holds under probabilistic
errors or prompt injection, and agent-authored
policy cannot weaken safety constraints;
and \emph{low overhead}, so enforcement
does not slow the agent's normal workload. To achieve them, two challenges exist:

\textbf{C1: High-level yet kernel-enforceable policy specification}
(\S\ref{sec:dsl}, \S\ref{sec:ifc}).
Generating policy at the intent level is too
ambiguous to enforce; generating at the
syscall level requires kernel expertise agents
may lack, produces
verbose rules that consume tokens and are error-prone.
The DSL must be
\emph{high-level} enough for agents to easily
express complex cross-event policy
with minimal tokens, yet
\emph{concrete} enough to compile to kernel constraints
and be enforced with low overhead.

\textbf{C2: Authority-safe agent programmability}
(\S\ref{sec:authority}). Agents need to adapt richer IFC policies 
at runtime (a sub-agent may need
a narrower scope, or the current task may require a new constraint),
but must not weaken constraints.

\subsection{Threat Model}

Following the principle of the self-restriction model of
Landlock~\cite{landlock} and Capsicum~\cite{watson2010capsicum}, \sys{}
assumes the agent being enforced may be unsafe or compromised.
The adversary's influence is bounded by the enforced agent's normal capabilities;
the kernel enforcement engine and higher-authority policy blobs are
loaded by a privileged process before the agent starts and are not
writable by the agent. 

\sys{} provides two levels of assurance, by policy authority.
Higher-authority rules can carry a safety and security assurance
under our threat model. 
Agent-authored rules are only a compliance assistance under a
cooperative-but-fallible model: enforcement is deterministic, an
adversary controlling the agent at policy-generation time can only weaken
self rules; the worst case degrades to having no self-imposed rules,
and cannot change inherited ones. Given loaded rules, \sys{} enforces
(1): agent-authored rules 
never weaken inherited constraints; and
(2): every event
delivered to a configured
hook is checked (hooks: exec, open, read,
write, unlink, connect, fork, exit).
Whether a violation is caught
depends on \emph{policy coverage}.
Block effects are synchronous
(no TOCTOU gap); kill effects are
post-operation (the process is terminated 
before subsequent monitored
action); notify is observation-only.

The TCB is the eBPF engine and higher-authority blobs; the compiler and
runner are outside it (bugs weaken only the agent's own constraints).
\sys{} covers indirect execution paths (subprocesses, shell wrappers
) that produce exact hooked OS events; 
semantic equivalents bypass and file contents are not covered, 
and hook types are limited by implementation.
(e.g., a custom Git client avoids \texttt{exec "git"} rules, but its
connect/write syscalls remain visible).
The monitored domain is the agent's entire process tree.
Kernel compromise, \texttt{CAP\_BPF} compromise, side channels, and
content-level checks are out of scope.

\subsection{System Overview}

\sys{} addresses these challenges with three abstractions
(Figure~\ref{fig:architecture}): a policy DSL
(\S\ref{sec:dsl}, C1) with a labeled information-flow model
(\S\ref{sec:ifc}); and a layered authority model
(\S\ref{sec:authority}, C2).
Because enforcement operates at the OS kernel layer, \sys{} is
runtime-agnostic. When a rule fires, \sys{} returns semantic feedback: the rule name,
reason, and a remediation path.

\subsection{Policy DSL}
\label{sec:dsl}

The DSL bridges AI agent intent and enforceable system actions.
A policy contains \emph{sources} that introduce labels,
\emph{rules} that match labeled events at sinks, and optional
\emph{gates} that express cross-event ordering constraints. The
DSL restricts expressiveness to source/sink/gate patterns that compile
to fixed-size rodata, avoiding direct exposure of BPF hooks, maps, label
bits, or verifier constraints.

Sources bind labels to system objects: an \texttt{exec} source labels
processes that execute a matching binary; a \texttt{file} source labels
matching paths; a network source labels matching endpoints. Rules bind an
effect (\texttt{notify}, \texttt{block}, or \texttt{kill}) to an
operation (e.g. \texttt{exec}, \texttt{open}). Conditions express common agent
workflow constraints: a target allow-list, a lineage gate, or a temporal
\texttt{after~...~since~...} ordering gate. An \texttt{exits~$N$} qualifier
on the gate command requires the gating process to terminate with exit
code~$N$, distinguishing a successful run from a crashed or failing one.

\begin{figure}[t]
\begin{verbatim}
source AGENT = exec "claude"

# Per-event: from cloudflare/moltworker
rule no-delete-data:
  block unlink "/data/moltbot/**" if AGENT
  because "Never delete R2 data; the mount IS the bucket."

# Cross-event: from chenhg5/cc-connect
rule tests-before-commit:
  kill exec "git" "commit"
    if AGENT unless after exec "go" exits 0
      since write "**/*.go"
  because "Run `go test ./...` before committing."

# Cross-event notify: from vxcontrol/pentagi
rule regen-after-schema:
  notify exec "git" "commit"
    if AGENT unless after exec "gqlgen"
      since write "**/schema.graphqls"
  because "Re-run gqlgen after schema changes."
\end{verbatim}
\caption{Three \sys{} rules drawn from real projects: a per-event block,
a cross-event kill gate, and a cross-event notify gate.}
\Description{Code listing of three \sys{} rules from real projects.}
\label{fig:policy-example}
\end{figure}

Figure~\ref{fig:policy-example} illustrates the three main rule shapes.
The first is per-event: if the agent process deletes any file under the
data mount, the event is blocked immediately. The second is cross-event
with a kill effect: committing is
compliant only if the required test command ran and exited successfully
(\texttt{exits~0}) after the most recent source write. The third is also cross-event but uses a notify effect:
the agent is reminded, not blocked, when committing after a schema change
without re-running the code generator.

The DSL mainly targets the policy shapes that recur in agent instruction
files: prohibit a command, restrict writes to a scope, block a sink after
a sensitive read, and
route sensitive data through an approved tool. The compiler lowers these
patterns into fixed-size kernel tables; policy authors need not reason
about BPF maps, verifier constraints, or binary layout.

\subsection{Labeled Information-Flow Model}
\label{sec:ifc}

\sys{} represents policy state as labels on OS objects. Processes, files,
and network endpoints each carry a 64-bit label mask, enabling
cross-boundary flow policies (e.g., ``data read from \texttt{.env} must
not reach a network endpoint''). A source declaration adds labels when an
object matches the source pattern. Let $L(x)$ denote
the label mask of object $x$. Labels propagate through observed
system-level edges: 

\begin{itemize}
\item \textbf{fork}$(p \to c)$: $L(c) := L(c) \cup L(p)$.
\item \textbf{exec}$(p, f)$: $L(p) := L(p) \cup L(f) \cup S(f)$, where
$S(f)$ is the set of source labels matching $f$.
\item \textbf{open/read}$(p, f)$: $L(p) := L(p) \cup L(f)$.
\item \textbf{write}$(p, f)$: $L(f) := L(f) \cup L(p)$.
\item \textbf{unlink}$(p, f)$: no propagation; rule check only.
\item \textbf{connect}$(p, e)$: $L(e) := L(e) \cup L(p)$.
\end{itemize}

These transfer functions maintain a key invariant: if an object carries
label $\ell$, then information from a source tagged with $\ell$ may have
reached that object through observed execution edges. Rules check whether
an event's subject or target carries required labels, carries forbidden
labels, or satisfies a temporal gate.

\sys{} trades precision for 
low-overhead whole-session coverage, since content 
can be tracked by combining other tools when necessary 
(e.g. linter or static analyzer).
Object-level labels capture indirect, cross-event 
patterns that tool-call filters or per-event sandbox miss, 
without incurring the complexity of
byte-level taint tracking~\cite{kemerlis2012libdft}. 
For efficiency, labels further summarize history:
each rule checks masks and
bounded predicates at the current event. This keeps cross-event
enforcement without log reconstruction or
history trace queries.

Labels are monotonic: propagation adds labels but never removes them.
Once an object carries a label, subsequent consumers inherit it.
This ensures that cross-event constraints remain checkable for the
entire session without requiring explicit label cleanup.
As in Flume~\cite{flume} and HiStar~\cite{zeldovich2006histar}, labels
on objects are irrevocable by default. Declassification follows our authority 
model: the author of the rule holds the declassification
privilege and can disable the rules 
and clean the labels related to the rules.
Inherited safety labels and rules cannot be declassified.
The recommended mitigation for inherited labels is process-level
isolation.

\subsection{Layered Policy Authority}
\label{sec:authority}

\sys{} must let agents adapt policy at runtime without
weakening inherited safety constraints.
Following the self-restriction principle,
\sys{} extends monotonic
tightening in Landlock~\cite{landlock} and
Capsicum~\cite{watson2010capsicum} to a
\emph{domain hierarchy} over labeled IFC state that spans process trees.
A domain is the runtime policy boundary for a process tree:

\begin{small}
\begin{verbatim}
pid    -> domain
domain -> parent + policy(domain)
policy(D) = policy(parent(D)) + local(D)
\end{verbatim}
\end{small}

\noindent A child domain inherits all parent rules
and may add local rules, labels, or gates, but cannot remove, disable,
or weaken any inherited rule. The root domains correspond to
platform/operator and project-maintainer policy; the agent creates
session and sub-agent domains below them.

Runtime policy additions are submitted as precompiled deltas through an eBPF
userspace ring buffer, all submission permission checks 
remain in kernel. The eBPF checker derives the submitting domain from the
caller's pid-to-domain mapping and admits updates only to the
submitting domain or its descendants. A delta is accepted only if it
introduces fresh labels, binds new rules, or narrows scope, without
modifying inherited rules, labels, or effects; a delta cannot create
labels that satisfy an inherited exception gate unless higher-authority
policy explicitly defines that path. Accepted deltas are merged into
the domain's effective policy on the event path without parsing
configuration or resolving project context.

\section{Implementation}
\label{sec:implementation}

\sys{} consists of a Rust userspace compiler/runner (~3.2\,K\,LoC) 
and an eBPF
enforcement engine (~1.8\,K\,LoC of BPF C). The compiler parses the DSL,
lowers Boolean label expressions and path/network
patterns, and emits a fixed-size \texttt{\#[repr(C)]} configuration blob
consumed directly by the eBPF loader. Human-readable rule names and
reasons stay in userspace metadata; the kernel receives only compact rule
tables and emits rule identifiers when events match.

The eBPF engine attaches to process, file, and network hooks via
BPF-LSM for pre-operation enforcement (block effects) and tracepoints
for observation and post-operation termination (kill effects),
maintaining label state in per-object BPF maps.
A \texttt{sched\_process\_exit} tracepoint reads
\texttt{task\_struct.exit\_code} to arm exit-qualified gates only on
normal termination with the specified status. Userspace does not 
re-detect violations; it only formats
kernel-reported matches into semantic feedback. 

The runner (\texttt{\sys{} run -- <cmd>}) compiles and loads policy
before the target starts, seeds the agent label, and records
kernel-reported matches in a feedback file that agent hooks relay into
the model's next turn.
The authority checker runs entirely in the kernel eBPF engine. 
Each domain is a
BPF map entry storing a parent pointer, an inherited-rule mask, and an
inherited-label mask; pid-to-domain mappings are in a separate map.
Runtime deltas arrive through a ring buffer; the checker resolves 
the submitting domain via \texttt{bpf\_get\_current\_pid\_tgid()} 
and rejects any delta that clears an inherited-mask bit or modifies
 an inherited effect.
