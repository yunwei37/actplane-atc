\section{Design}
\label{sec:design}

\sys{} is a runtime-agnostic, 
programmable OS-level policy enforcement
system that provides the four requirements
identified above.
To achieve them, two challenges exist:

\textbf{C1: High-level yet kernel-enforceable policy specification}
(\S\ref{sec:dsl}, \S\ref{sec:ifc}).
The DSL should hide kernel complexity 
so agents can
write and modify deterministic policies
easily with minimal 
expertise about kernel, DSL and token cost;
most real policies are structurally 
simple (\S\ref{sec:eval:rq1}).
Yet generating at the intent level is 
too ambiguous to enforce,
and generating at the syscall level 
is costly and error-prone.
The DSL needs to be
\emph{high-level} enough for agents to easily
express policy with minimal cost, yet
\emph{concrete} enough to compile to kernel constraints
and be enforced with low overhead.

\textbf{C2: Policy-isolated agent programmability}
(\S\ref{sec:authority}). Agents need to adapt IFC policies
at runtime (a sub-agent may need
a narrower scope, or the current task
may require a new constraint),
but must not affect other agents' policies
or weaken inherited constraints.


\sys{} addresses these challenges with three abstractions
(Figure~\ref{fig:architecture}). A policy DSL
(\S\ref{sec:dsl}) that lets agents express policy intent
in a representative form close
to natural language. A compiler
lowers the DSL into a fixed-size kernel config.
A labeled information-flow state machine model
(\S\ref{sec:ifc}) that tracks cross-event 
state at OS level (C1). Layered
policy domains (\S\ref{sec:authority})
that let agents
adapt policy at runtime without weakening
inherited constraints (C2).
When a rule fires, \sys{} returns semantic feedback: the rule name,
reason, and a remediation path.

\subsection{Policy DSL}
\label{sec:dsl}

The DSL lets agents express policy intent
as enforceable system-level constraints. We
observe that
OS-enforceable directives from the empirical
study share a common structure: they constrain
operations from a source
(a process, a file read) reaching a target
(a file write, a network endpoint).
Per-event directives match one operation;
cross-event directives add history conditions.
The DSL thus models both as source-to-target
constraints. Thus, we map the directive to
a \emph{rule} in the DSL.

Each \emph{rule} encodes one OS-enforceable directive:
it specifies a \emph{source} (which process or data),
a \emph{target} operation, an \emph{effect} to apply
when the source reaches the target,
an optional temporal \emph{gate},
and a \emph{reason} for semantic feedback.

\emph{Sources} tag system objects:
an \texttt{exec} source tags processes
that execute a matching binary;
a \texttt{file} source tags matching paths;
a network source tags matching endpoints.
A rule checks a source condition at a
target operation
(e.g.\ \texttt{exec}, \texttt{open})
and applies one of three \emph{effects}:
\texttt{block} synchronously
denies the syscall pre-operation
(no TOCTOU gap; the agent continues
and can retry a different action),
\texttt{kill} terminates the offending
process post-operation
(preventing it from trying alternative channels),
and \texttt{notify} only reports without
preventing execution.
For example, \texttt{block write} suits
an agent writing to a
wrong directory, while \texttt{kill write}
suits a compromised agent writing
outside its sandbox that should not continue.
\texttt{block} and \texttt{kill}
enforce constraints (``do not X'')
while \texttt{notify} observes
instructions (``do X'')~\cite{zhang2026guardrails}.
A rule may also specify a target allow-list
to restrict which paths or endpoints
it applies to.

Source labels record that an event occurred
but not when; cross-event directives like
``run tests before committing'' need ordering.
\emph{Gates} fill this gap by tracking
temporal ordering, such as whether a
test ran after the most recent edit.
Gate conditions include a lineage gate 
or a temporal
\texttt{after~...~since~...} ordering gate. 
An \texttt{exits~$N$} qualifier
on the gate command requires the gating process 
to terminate with exit
code~$N$, distinguishing a successful run from
a crashed or failing one.

\begin{verbatim}

# Per-event: from cloudflare/moltworker
rule no-delete-data:
  block unlink "/data/moltbot/**" if AGENT
  because "Never delete R2 data;

# Cross-event: from chenhg5/cc-connect

  kill exec "git" "commit"
    if AGENT unless after exec "go" "test" exits 0

rule regen-after-schema:

\end{verbatim}

\subsection{Labeled Information-Flow}
\label{sec:ifc}

Enforcing the DSL's source-to-target constraints
requires tracking which sources have influenced
which objects across the session.
\sys{} implements this with a \emph{labeled
information-flow} model.
Following classical
IFC~\cite{flume,zeldovich2006histar},
the model is a \emph{state machine}: each
tracked object carries state that is updated
at system events, so rules are checked
without reconstructing the history log.

Rules need to check whether the subject
or target has been influenced by certain sources.
To support this, each process, file, and
network endpoint carries a set of \emph{labels}.
Each source declaration in the DSL defines
a label; when a system object matches
a source pattern, the label is added.
At each event, the engine checks whether
the subject's and target's labels satisfy
a rule's conditions (required labels present,
forbidden labels absent, gate satisfied)
and applies the rule's effect.

Cross-event rules must also reflect sources
from earlier events, so labels 
need to \emph{propagate}
along OS data-flow edges
(e.g. fork, exec, read, write, connect).
This maintains a key \emph{invariant}: if an object
carries label~$\ell$, then information from
a source tagged with~$\ell$ may have reached
it through observed edges.

For example, data read from \texttt{.env}
acquires its source label; if that process
later connects to an external endpoint,
a rule forbidding that label at network
targets fires and blocks the connection.

To ensure no history is lost and cross-event
constraints remain checkable for the entire
session, labels are \emph{monotonic}:
propagation adds labels but never removes them.
Once an object carries a label, subsequent
consumers inherit it.
As in Flume~\cite{flume} and
HiStar~\cite{zeldovich2006histar}, labels
are irrevocable by default;
declassification is delegated to the
domain hierarchy (\S\ref{sec:authority}).

\subsection{Threat Model}
\sys{} assumes the enforced agent, 
which starts after
\emph{higher-authority} policy is loaded and
may author additional rules at runtime,
may be unsafe or compromised. This
follows the principle of
self-restriction in Landlock~\cite{landlock}
and Capsicum~\cite{watson2010capsicum}.
The adversary's influence is bounded by
the enforced agent's normal capabilities;
the kernel enforcement engine and
higher-authority policy are
loaded by a privileged process before
the agent starts and are not
writable by the agent.
The \emph{trust boundary} is temporal:
any pre-defined rule loaded into the
kernel before the enforced agent starts
is higher-authority,
regardless of whether a human or a separate
policy-generation agent authored it.

\sys{} provides two levels of
assurance, by policy authority.
Higher-authority rules can
carry a safety and security assurance
under our threat model.
Agent-authored runtime rules are only a
compliance assistance under a
cooperative-but-fallible model:
enforcement is deterministic, an
adversary controlling the agent at
policy-generation time can only weaken
self rules; the worst case degrades to
having no self-imposed rules,
and cannot change inherited ones.
Given loaded rules, \sys{} enforces
(1): agent-authored rules
never weaken inherited constraints; and
(2): every event
delivered to a configured
hook is checked.
Whether a violation is caught
depends on \emph{policy coverage}.
Block is synchronous (no TOCTOU gap);
kill is post-operation; notify is
observation-only.

The TCB is the kernel enforcement
engine and higher-authority blobs;
the compiler and
runner are outside it (bugs weaken
only the agent's own rules).
\sys{} covers indirect execution
paths (subprocesses, shell wrappers)
that produce exact hooked OS events;
semantically equivalent operations
and file contents are not covered,
and hook types are limited by implementation
(e.g., a custom Git client avoids
\texttt{exec "git"} rules, but its
connect/write syscalls remain visible).
The policy enforcement covers the agent's
entire process tree.
Kernel compromise, \texttt{CAP\_BPF}
compromise, side channels, and
content-level checks are out of scope.

\subsection{Layered Policy Domains}
\label{sec:authority}

Agents need to adapt policy at runtime
(a sub-agent may need a narrower scope),
but must not weaken constraints inherited
from higher authority.

To support this, \sys{} introduces 
\emph{policy domain}.
A \emph{domain} is the runtime policy boundary
for a process tree; it governs rule scope
and mutability, not resource access.
Following the self-restriction principle,
\sys{} extends monotonic
tightening in Landlock~\cite{landlock} and
Capsicum~\cite{watson2010capsicum} to a
\emph{domain hierarchy} over labeled IFC state
that spans process trees:

\begin{small}
\begin{verbatim}
pid    -> domain
domain -> parent + policy(domain)
policy(D) = policy(parent(D)) + local(D)
\end{verbatim}
\end{small}

\noindent A child domain inherits 
all parent rules
and may add local rules, labels, or gates,
 but cannot remove, disable,
or weaken any inherited rule.
Rules added by an agent take effect only 
within its domain and descendants, so
multiple agents on the same system
can enforce different policies.

To support runtime policy updates and
declassification, \sys{}
requires an \emph{authority checker} in
the kernel:
runtime policy additions are submitted
as \emph{deltas} through a ring buffer,
all
permission checks
remain in the kernel. A delta is
accepted only if it
introduces fresh labels, binds new rules,
or narrows scope, without
modifying inherited rules, labels,
or effects;
a delta cannot create
labels that satisfy an inherited exception
gate unless higher-authority
policy explicitly defines that path.
Accepted deltas are merged into
the domain's effective policy.
Declassification follows the domain hierarchy:
the author of a rule holds the declassification
privilege. It can disable rules
and clear labels within its own domain.
Inherited labels and rules cannot be
declassified;
the recommended mitigation for inherited
labels is process-level isolation.

\section{Implementation}
\label{sec:implementation}

\sys{} consists of a Rust userspace compiler/runner (~3.2\,K\,LoC) 
and an eBPF
enforcement engine (~1.8\,K\,LoC of BPF C). The compiler parses the DSL,
lowers Boolean label expressions and path/network
patterns, and emits a fixed-size \texttt{\#[repr(C)]} configuration blob
consumed directly by the eBPF loader. Human-readable rule names and
reasons stay in userspace metadata; the kernel receives only compact rule
tables and emits rule identifiers when events match.

The eBPF engine attaches to process, file, and network hooks
(exec, open, read, write, unlink, connect, fork, exit) via
BPF-LSM for pre-operation enforcement (block effects) and tracepoints
for observation and post-operation termination (kill effects),
maintaining label state as 64-bit bitmasks in per-object BPF maps;
propagation is a single bitwise OR.
A \texttt{sched\_\allowbreak{}process\_\allowbreak{}exit} tracepoint reads
the task exit code to arm exit-qualified gates only on
normal termination with the specified status. Userspace does not 
re-detect violations; it only formats
kernel-reported matches into semantic feedback. 
The engine supports up to 128 concurrent rules, sized to cover the
largest observed repository (66 OS-enforceable directives) with margin
for layered domain composition. 
The eBPF checker derives the submitting 
domain from the
caller's pid-to-domain mapping 
and admits updates only to the
submitting domain or its descendants.

The runner (\texttt{\sys{} run -- <cmd>}) compiles and loads policy
before the target starts, seeds the agent label, and records
kernel-reported matches in a feedback file that agent hooks relay into
the model's next turn.
The authority checker runs entirely in the kernel eBPF engine. 
Each domain is a
BPF map entry storing a parent pointer, an inherited-rule mask, and an
inherited-label mask; pid-to-domain mappings are in a separate map.
Runtime deltas arrive through a ring buffer; the checker resolves 
the submitting domain via \texttt{bpf\_get\_\allowbreak{}current\_\allowbreak{}pid\_tgid()} 
and rejects any delta that clears an inherited-mask bit or modifies
 an inherited effect.
Extending \texttt{op\_pattern} and temporal gates remains future work.
