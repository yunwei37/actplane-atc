\section{Evaluation}
\label{sec:evaluation}

We evaluate \sys{} along three axes. RQ1 measures end-to-end policy
compliance on directive-derived rules from the empirical corpus;
RQ2 quantifies per-event and end-to-end overhead; RQ3 tests
repository-grounded coding tasks on an OctoBench subset with the official
judge.

\begin{description}
\item[RQ1 (Policy compliance)] Compared with prompt-filter, tool-layer,
  and feedback-ablation baselines, does \sys{} improve policy
  compliance for directive-derived rules across direct and bypass
  execution paths?
\item[RQ2 (Overhead)] What is the per-event and end-to-end overhead of
  \sys{}'s label propagation and rule checking?
\item[RQ3 (Repository-grounded feedback)] On a scaffold-aware
  repository-grounded coding benchmark, does \sys{}'s OS-level
  enforcement and semantic feedback improve official policy compliance?
\end{description}

\subsection{Experimental Setup}
\label{sec:eval:setup}

\paragraph{Hardware and kernel.}
All experiments run on a single machine with an Intel Core Ultra~9~285K
CPU (24~hardware cores), 125\,GiB RAM, Linux~6.15.11, ext4 filesystem,
and libbpf~1.x with \texttt{bpf\_loop} support. The live enforcement
runs require BPF-LSM active (\texttt{bpf} in
\texttt{/sys/kernel/security/lsm}); the RQ2 overhead artifacts record
the exact LSM state used for each performance run.

\paragraph{Agent environment.}
% TODO: Fill in agent versions.
The active AI agent harnesses are Claude Code (via the \sys{} feedback
hook) and OpenAI Codex CLI (via tool-error feedback). Exact versions are
recorded in per-run metadata.

\paragraph{Compared systems.}
RQ1 uses four runtime policy enforcement systems on the same
trace-conditioned AI agent harness: a prompt-based action filter, a
tool-input regex policy, \sys{} without structured corrective feedback, and
full \sys{}. The prompt and regex baselines instantiate two common
runtime enforcement layers: LLM/prompt-based rails over proposed
actions~\cite{rebedea2023nemo,xiang2025guardagent} and deterministic
tool-call policies over tool names and arguments in the style of
AgentSpec~\cite{wang2025agentspec}. The opaque ablation isolates the value
of semantic feedback on top of OS-level observation.

\subsection{Rule Set Construction}
\label{sec:eval:ruleset}

The empirical study identifies 607~OS-enforceable directives:
392~per-event and 215~cross-event. RQ1 uses this subset because these
directives can be represented at process, file, or network hooks.
Semantic-only and content directives are excluded: the former have no
system-observable counterpart; the latter require file-content inspection
rather than OS-level IFC. The evaluation pipeline enforces strict
separation of concerns across four actors.

\paragraph{Pipeline overview.}
The pipeline has five stages with strict actor separation
(Figure~\ref{fig:rq1-pipeline}).
A \emph{translation agent} reads the sampled directive text, the
project's \texttt{CLAUDE.md}, the cloned repository, and the DSL
reference to produce a rule file. The agent iteratively invokes the
\sys{} compiler as a tool: if a generated rule fails to compile, the
agent observes the compiler diagnostic, revises the rule, and retries
until compilation succeeds. This closed-loop process ensures that every
sampled directive yields a syntactically valid, compiled DSL rule; the
translation agent cannot see traces or ground truth.
A separate trace generator reads the directive and project structure to
produce five challenge trace families per statement, each with a
ground-truth header; it cannot see the rule file, preventing circular
bias. The families cover allowed-effect compliant behavior, lookalike
compliant behavior, directly visible violations, script-mediated
violations, and opaque fixture violations. Script-mediated traces split
script authoring and script execution across separate tool calls; opaque
traces expose only a neutral helper entrypoint while the real side effect
is visible at runtime.
The eval harness replays each trace under each system, collects the
system response, and issues bounded tool-action turns to a tested LLM, a
smaller model standing in for the agent.
Finally, a scorer compares the tested LLM's decision against
ground truth and classifies the outcome as TP, TN, FP, or FN.
Because the translation agent and the trace generator are different
actors, translation blind spots do not hide in traces; because the tested
LLM never sees ground truth, it acts like a real agent.
Runtime artifacts are frozen before execution from the same directive and
project context.
Figure~\ref{fig:policy-example} shows representative per-event and
cross-event \sys{} rules.

\subsection{RQ1: End-to-End Policy Compliance}
\label{sec:eval:rq1}

\begin{figure}[t]
\centering
\includegraphics[width=\columnwidth]{figures/rq1-pipeline.pdf}
\caption{RQ1 evaluation pipeline: three enforcement paths from
natural-language rule to agent-level decision.}
\Description{Flow diagram showing prompt-filter, tool-regex, and
\sys{} paths through translation, enforcement, and judging.}
\label{fig:rq1-pipeline}
\end{figure}

\paragraph{Method.}
RQ1 uses a trace-conditioned benchmark derived from the OS-enforceable
directive corpus (Figure~\ref{fig:rq1-pipeline}). It is a system-effect challenge benchmark: each trace
exercises whether a runtime policy enforcement system connects an
intent-level directive to concrete tool and OS effects. The current
paper-facing sample contains 38 statements.
The sample is manifest-selected: we select 2--3 statements from each of 15 repositories to cover both
per-event and cross-event rules, project- and task-context requirements,
and common repository policy themes. The final manifest contains 20
per-event-like and 18 cross-event-like rules. No system outcome is used to
select statements or repair policy artifacts.
For each statement we construct five manifest-listed traces: two compliant
challenge families (allowed-effect and lookalike) and three violation
families (visible, script-mediated, and opaque-fixture). This yields
190 traces and 760 system--trace cells across four systems: prompt-filter,
tool-regex, \sys{} without structured feedback (\textsc{ActPlane-opaque}),
and full \sys{}.

RQ1 is an end-to-end runtime enforcement measurement. Given the same
natural-language directive and the same trace-conditioned agent context,
each system must connect the directive to the proposed action or resulting
system effects, intervene on violations, and, where feedback is available,
steer the agent toward a policy-correct outcome. The four systems differ in
how the directive enters the harness. \textsc{Prompt-filter} has no
translated policy artifact: before each proposed tool action, a separate
stateless LLM classifier receives the natural-language directive and the
proposed action and returns allow, report, or block feedback.
\textsc{Tool-regex} uses a per-case regex artifact generated before
execution from the directive and project context, then checks raw Agent SDK
tool names and arguments. \textsc{ActPlane-opaque} uses the generated
\sys{} DSL policy below the tool layer but hides structured corrective
feedback from the agent. Full \sys{} uses the same generated OS policy and
returns the matched rule reason and remediation. All generated policy
artifacts are frozen before execution and are not repaired after outcomes.

Each trace is a JSONL transcript with a ground-truth header followed by a
user prompt and concrete tool calls that establish the repo state. The
trace files do not store static tool results; during replay the harness
executes each tool call on a real copy-on-write repository and records the
actual result. The tested agent then continues for a bounded number of real
Agent SDK tool steps. All systems use the same trace, model, Docker
copy-on-write repo execution, and trajectory judge. Ground truth and policy
artifacts are not shown to the tested agent.

A separate LLM trajectory judge assigns TP, TN, FP, or FN from the raw
runner result. TP means a valid violation trace received an
enforcement-visible report, block, failure, intervention, or agent
awareness; TN means a valid compliant trace completed without wrongful
enforcement intervention; FP means a compliant trace was wrongly blocked,
reported, failed, or steered away; FN means a violation trace was missed
or left without enforcement-visible awareness. We report Decision
Compliance Rate (DCR), the fraction of trace-conditioned decision points
whose final outcome is policy-correct under the selected runtime policy
enforcement system: \(\mathrm{DCR}=(\mathrm{TP}+\mathrm{TN})/(\mathrm{TP}+\mathrm{TN}
+\mathrm{FP}+\mathrm{FN})\). DCR is end-to-end: it includes policy
translation, runtime enforcement, feedback delivery, and agent recovery;
it is not detector accuracy in isolation or unguided agent compliance.
For this snapshot, each
(\textit{system}, \textit{statement}, \textit{trace}) cell is selected from
the latest non-stale run for the current five-family manifest; all 760
runner cells and all 760 trajectory-judge cells are present, with no judge
parse errors.

\sys{} reaches a DCR 27.4 percentage points higher than prompt-filter and
30.5 points higher than tool-regex. The difference comes primarily from
violation recall. Full \sys{} reduces FNs to 28, compared with 70 for
prompt-filter and 76 for tool-regex, because it observes runtime side
effects below the tool layer and can detect script- and fixture-mediated
violations that are not available to prompt or raw tool-input filters.
It also reduces compliant-trace overblocking relative to the baselines,
though this margin is smaller: full \sys{} has 18 FPs, compared with 28 for
both prompt-filter and tool-regex.

The feedback ablation isolates the value of semantic recovery information.
\textsc{ActPlane-opaque} has very few false positives, but it misses many
violations (87 FNs) because the agent receives no structured corrective
payload after enforcement. Full \sys{} trades some additional compliant-trace
interventions for a much larger FN reduction, increasing correct responses
from 102 to 144. Thus, the result is not simply that kernel enforcement can
stop more effects; model-facing feedback is part of the end-to-end
policy-enforcement system.

\subsection{RQ2: Overhead}
\label{sec:eval:rq2}

\subsubsection{Macrobenchmarks (End-to-End Workloads)}

We measure end-to-end overhead on two deterministic workloads under
no-hit \sys{} configurations. The agent trace suite replays
20~corpus-derived traces (68~tool actions, 20~Bash subprocesses). The
Linux kernel build compiles \texttt{defconfig}~+~\texttt{vmlinux} with
\texttt{make~-j24} in a clean output directory. Each workload runs three
trials per configuration. Fixed workloads eliminate LLM inference variance
and isolate \sys{}'s runtime cost.

\begin{figure}[tb]
\centering
\includegraphics[width=\columnwidth]{figures/rq2_macro_overhead.png}
\caption{End-to-end overhead normalized to native execution.}
\Description{Grouped bar chart of AP-32 and AP-100 overhead for agent
trace replay and Linux build.}
\label{fig:rq2-macro}
\end{figure}

\subsubsection{Microbenchmarks (Per-Syscall Latency)}

We measure \sys{}'s per-event latency for six syscall types
(\texttt{fork}, \texttt{exec}, \texttt{open}, \texttt{read},
\texttt{write}, \texttt{connect}) under five no-hit configurations:
native, and \sys{} with 1, 10, 32, and 100 active rules (AP-$N$ denotes
\sys{} with $N$ rules). Each configuration $\times$ syscall type runs
10K--100K iterations pinned to a single CPU core. For each trial we
compute p50 and p99 latencies; Table~\ref{tab:micro-latency} reports
the median across seven trials.

\paragraph{Measured operations.}
\texttt{fork}: \texttt{fork()} + parent \texttt{waitpid()}.
\texttt{exec}: \texttt{fork()} + child \texttt{execve("/bin/true")} +
parent \texttt{waitpid()}.
\texttt{open}: \texttt{open(path, O\_RDONLY|O\_CLOEXEC)} with
\texttt{close()} outside the timed region.
\texttt{read}: \texttt{read(fd, buf, 4096)} from an already-open file.
\texttt{write}: \texttt{write(fd, buf, 4096)} to an already-open temp file.
\texttt{connect}: UDP \texttt{connect(127.0.0.1:9)} on an
already-created socket.

\paragraph{Results.}

\centering

Values are medians across seven trials.}
\label{tab:micro-latency}
\begin{tabular}{lrrrr}
\toprule

\midrule
\texttt{fork}    & 48.94 & 74.05 & 69.33 & 178.01 \\

\texttt{write}   & 0.27 & 0.79 & 0.84  & 1.59 \\
\texttt{connect} & 0.58 & 1.98 & 3.17  & 3.90 \\
\midrule

\bottomrule
\end{tabular}
\end{table}

Table~\ref{tab:micro-latency} reports per-call latencies ordered by
operation weight in a typical agent session. \texttt{fork} and
\texttt{exec} dominate per-call cost in absolute terms but occur
infrequently; their overhead ratio is modest (1.3--1.5$\times$) because
process-management and scheduler latency dominates the baseline.
File operations (\texttt{open}, \texttt{read}, \texttt{write}) and
\texttt{connect} are fast in the native case; \sys{} adds a path-hash
lookup and rule scan on each. The weighted average across all six
operations (weighted by empirical frequency in the agent-trace workload)
is 4.68\,\textmu s at AP-100 p50, a 4.1$\times$ multiplier over the
native weighted average.

Despite per-call multipliers on individual fast-path operations, the
macrobenchmark impact remains low (1.9\%--8.4\%) because the absolute
added cost per call is small relative to the computation, I/O, and LLM
inference time that dominates agent workloads. A single LLM inference
turn (2--10\,s) exceeds the cumulative \sys{} overhead of an entire
tool-call's syscall sequence by 5--6 orders of magnitude.

All overhead measurements use no-hit configurations, which exercise
steady-state label propagation and rule scanning without violation
reporting or userspace feedback formatting.

\subsection{RQ3: Repository-Grounded Policy Compliance (OctoBench)}
\label{sec:eval:rq3}

\paragraph{Goal.}
Unlike RQ1, which isolates single decision points, RQ3 evaluates
\sys{} on complete coding-agent tasks in real repository environments,
scored by the official OctoBench whole-case checklist judge.

\paragraph{Benchmark and subset.}
OctoBench~\cite{octobench} is a scaffold-aware repository-grounded
benchmark with 217~Dockerized tasks spanning system prompts, tool
schemas, project files (\texttt{CLAUDE.md}, \texttt{AGENTS.md}), and
user queries. We use the official evaluator unchanged. Because \sys{}
cannot observe purely semantic checks (e.g., tone), we select a 20-task
OS-observable subset. A task is eligible when at least one checklist item maps to a
concrete command, file operation, generated artifact, or tool-execution
constraint that can be expressed as a case-specific policy before
execution. The selected subset spans \texttt{CLAUDE.md},
\texttt{AGENTS.md}, and user-query policy sources, two agent scaffolds,
and seven Docker images. Pure style, tone, and natural-language-only
cases are excluded.
Each task runs under three conditions: baseline (no enforcement),
tool-regex (case-specific tool-call hooks), and \sys{} (OS-level hooks
plus semantic feedback).

\paragraph{Metrics.}
The primary metric is official OctoBench reward (fraction of checklist
policies judged satisfied). We additionally report three diagnostic
submetrics from the same checklist results: \emph{user-query reward}
(user-task checks only), \emph{implementation/test reward}
(implementation, testing, configuration, and modification checks), and
\emph{compliance reward} (compliance-typed checks). All submetrics use the
official LLM-based checklist judge.

\begin{figure}[tb]
\centering
\includegraphics[width=\columnwidth]{figures/octobench_rq3.png}
\caption{OctoBench 20-task subset: reward breakdown by system.}
\Description{Grouped bar chart of baseline, tool-regex, and \sys{}
reward components.}
\label{fig:octobench-rq3}
\end{figure}

\paragraph{Results.}
On the 20-task subset, official reward is 0.788 (baseline), 0.818
(tool-regex), and 0.888 (\sys{}): a 10.0-point gain over baseline and
7.0~points over tool-regex. The improvement extends beyond generic
compliance: user-query reward rises by 28.2~points and
implementation/test reward by 25.0~points over baseline. These results
support the claim that OS-level enforcement with semantic feedback
improves policy compliance in repository-grounded coding agents. We note
that OctoBench uses an LLM checklist judge rather than deterministic test
scripts, so we do not claim deterministic task completion.

\section{Related Work}
\label{sec:related}

\textbf{In-kernel provenance and information flow.}
Whole-system provenance systems such as PASS~\cite{muniswamy2006pass},
Hi-Fi~\cite{pohly2012hifi}, LPM~\cite{bates2015lpm}, and
CamFlow~\cite{pasquier2017camflow} record audit-grade provenance
graphs via LSM hooks but do not compile agent-oriented policy or return
semantic feedback. CamQuery~\cite{pasquier2018camquery} is the closest
prior system: it evaluates provenance queries inline in the kernel,
propagates labels across OS objects, and can deny matching operations.
\sys{} shares this label-propagation model but targets cooperative AI
agents rather than adversarial intrusions, compiles an agent-facing
source/sink DSL rather than general provenance queries, and returns
semantic feedback to the matching actor. Dynamic taint-tracking
systems~\cite{enck2010taintdroid,clause2007dytan,kemerlis2012libdft,yin2007panorama}
operate at byte or instruction granularity for vulnerability detection;
\sys{} operates at object granularity, trading precision for low-overhead
whole-session coverage.

\textbf{eBPF enforcement and agent policy systems.}
BPF-LSM~\cite{singh2019krsi,linuxbpflsm} lets eBPF programs attach to
security hooks and return allow/deny verdicts; \sys{} uses it as its
pre-operation enforcement backend. Contemporary eBPF tools such as Tetragon,
Tracee, and eBPF-PATROL~\cite{ciliumtetragon,aquasecuritytracee,ghimire2025ebpfpatrol} provide
per-event predicates or single-channel lineage flags; \sys{} propagates
multi-label information flow across channels and checks cross-event
conditions. At the agent layer, programmable rails and guard agents
check prompts or action trajectories~\cite{rebedea2023nemo,xiang2025guardagent},
and AgentSpec~\cite{wang2025agentspec} enforces deterministic guards at
the tool-call boundary; \sys{} complements them below the tool API, where
shell-outs and subprocesses remain visible. FIDES and
CaMeL~\cite{costa2025fides,debenedetti2025camel} apply typed IFC within
the agent loop but do not expose a layered authority stack that separates
platform, project, and agent policy; Flume~\cite{flume} provides
decentralized labels with explicit capabilities but requires
per-application integration rather than a compile-time authority
hierarchy. \sys{} applies information-flow reasoning at the OS boundary
with a layered authority model, where observation persists across context
windows and cannot be circumvented by subprocess indirection.

\textbf{Agent instruction files.}
Recent studies characterize \texttt{CLAUDE.md} and \texttt{AGENTS.md}
files along dimensions of content taxonomy, maintenance practices, and
task
efficiency~\cite{chatlatanagulchai2025manifests,chatlatanagulchai2025agentreadmes,
santos2025claudeconfig,lulla2026agentsmd}. These works classify at file or
section-heading granularity. Our empirical study
(\S\ref{sec:empirical}) classifies at statement level along four axes,
specifically asking which instructions require cross-event enforcement
and which need project or task context to instantiate.

\section{Conclusion}
\label{sec:conclusion}

An empirical study of 64~projects shows that agent instruction files are
behavioral policies whose enforcement demands OS-level observation,
cross-event state, and agent-provided context. \sys{} addresses this by
letting agents author labeled information-flow policies that an eBPF/BPF-LSM
engine enforces below the tool layer, returning semantic feedback when rules
match. A layered authority model prevents agents from weakening
higher-authority constraints. \sys{} does not cover semantic-only
directives (17\%) or content-inspection directives (38\%) that require a
linting layer, and the cooperative threat model excludes adversarial
evasion. Extending coverage to content-level checks and multi-agent
domains are natural next steps.
