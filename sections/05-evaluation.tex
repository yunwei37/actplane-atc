\section{Evaluation}
% \section{评估}
\label{sec:evaluation}

We evaluate \sys{} along five research questions (RQ1--RQ5, distinct from the empirical E-RQs in \S\ref{sec:empirical}).
% 我们沿五个研究问题（RQ1--RQ5，与 \S\ref{sec:empirical}中的实证 E-RQ 不同）评估 \sys{}。

\begin{description}
\item[RQ1 (DSL coverage)] Can a translation agent generate \sys{} policies for all 607 OS-enforceable policies?
% \item[RQ1（DSL 覆盖）] 一个翻译智能体能否为所有 607 条操作系统可执行策略生成 \sys{} 规则？
\item[RQ2 (Policy engine effectiveness)] Compared with baselines, does \sys{} improve policy compliance for rules from our empirical study across direct and indirect execution paths?
% \item[RQ2（策略引擎有效性）] 与基线相比，\sys{} 是否提高了来自我们实证研究的规则在直接和间接执行路径上的策略合规性？
\item[RQ3 (Overhead)] What is the per-event and end-to-end overhead of \sys{}'s label propagation and rule checking?
% \item[RQ3（开销）] \sys{} 的标签传播和规则检查的每事件和端到端开销是多少？
\item[RQ4 (Real-world coding tasks)] On coding-agent tasks in real repositories, does \sys{} improve policy compliance and task success rate?
% \item[RQ4（真实世界编码任务）] 在真实仓库中的编码智能体任务上，\sys{} 是否提高了策略合规性和任务成功率？
\item[RQ5 (Safety beyond coding)] On a safety benchmark covering data handling, system administration, and workplace tasks, do \sys{}'s agent-generated policies, loaded in a higher-authority domain, prevent unsafe agent behaviors?
% \item[RQ5（编码之外的安全性）] 在涵盖数据处理、系统管理和工作场所任务的安全基准上，\sys{} 的智能体生成策略，作为更高权限域加载，是否能阻止不安全的智能体行为？
\end{description}

\subsection{RQ1: DSL Generation Cost and Coverage}
% \subsection{RQ1：DSL 生成成本和覆盖}
\label{sec:eval:rq1}

The empirical study identifies 607 OS-enforceable policies, and RQ1 evaluates whether an agent can translate all of them into compilable \sys{} rules.
% 实证研究识别了 607 条操作系统可执行策略，RQ1 评估智能体能否将它们全部翻译为可编译的 \sys{} 规则。
A Codex agent~\cite{openai2025codex} with GPT-5.5~\cite{openai2026gpt55} reads each policy, the repository context, and the DSL reference, then writes the corresponding \sys{} rule; the runner invokes the compiler and retries once on syntax errors to allow minor self-correction.
% 一个 Codex 智能体~\cite{openai2025codex}配合 GPT-5.5~\cite{openai2026gpt55}读取每条策略、仓库上下文和 DSL 参考，然后编写相应的 \sys{} 规则；运行器调用编译器，语法错误时重试一次以允许小幅自我修正。
The translator compiled rules for all 607 policies on the first or second attempt, with only 2 requiring a retry (0.3\%), demonstrating that the DSL is accessible to current-generation LLMs.
% 翻译器在首次或二次尝试中为全部 607 条策略编译了规则，仅 2 条需要重试（0.3\%），表明该 DSL 对当代 LLM 是可访问的。
To validate correctness, we apply the same methodology as the empirical study (\S\ref{sec:empirical}): two independent LLM agents cross-check all policies, and human annotators review 100 randomly selected ones.
% 为验证正确性，我们采用与实证研究（\S\ref{sec:empirical}）相同的方法论：两个独立的 LLM 智能体交叉检查所有策略，人工标注者审查 100 条随机选择的策略。

Agent-based translation is orders of magnitude cheaper than manual authoring. The full 607-policy corpus cost \$0.028 per policy (1.7M input tokens, 177k output), compared with roughly \$11 per rule at typical US software-engineer rates.
% 基于智能体的翻译比人工编写便宜几个数量级。全部 607 条策略语料库每条成本 \$0.028（1.7M 输入 token，177k 输出），而按典型美国软件工程师费率约每条规则 \$11。
The translation completed in 34 minutes using 7 subagents (4 parallel), expanding 607 policies into 1{,}283 rule lines.
% 翻译使用 7 个子智能体（4 个并行）在 34 分钟内完成，将 607 条策略扩展为 1,283 条规则行。

\noindent\textbf{Most policies are structurally simple, making them amenable to agent generation.}
% \noindent\textbf{大多数策略在结构上是简单的，使其适合智能体生成。}
74\% have at most two enforcement clauses, and even the 95th percentile stays under 152 tokens (mean 71, OpenAI \texttt{o200k\_base} tokenizer).
% 74\% 最多有两个执行子句，即使第 95 百分位也在 152 token 以下（均值 71，OpenAI \texttt{o200k\_base} 分词器）。
Cross-event policies are slightly longer (mean 86 tokens) than per-event ones (62), but both remain well within single-prompt context limits.
% 跨事件策略略长（均值 86 token）于单事件策略（62），但两者都在单提示上下文限制范围内。

\noindent\textbf{The corpus exercises most DSL features, validating the language's expressiveness.}
% \noindent\textbf{该语料库运用了大多数 DSL 特性，验证了语言的表达力。}
Effects skew toward observation, with 66\% of clauses being notify, 29\% block, and only 5\% kill, reflecting that most policies monitor rather than prevent.
% 效果偏向观察，66\% 的子句是通知，29\% 阻止，仅 5\% 终止，反映出大多数策略监控而非阻止。
Hooks concentrate on code execution (60\% exec) and file mutation (37\% write), with network and cleanup operations under 1\% each.
% 钩子集中在代码执行（60\% exec）和文件变更（37\% write），网络和清理操作各占不到 1\%。

Cross-event features see substantial use: 28\% of policies use an \texttt{after/since} temporal gate, and 214 use \texttt{unless} to encode exceptions.
% 跨事件特性使用广泛：28\% 的策略使用 \texttt{after/since} 时间门控，214 条使用 \texttt{unless} 编码例外。
The 95 policies (15.7\%) that define file sources demonstrate information-flow tracking beyond simple agent provenance.
% 定义文件源的 95 条策略（15.7\%）展示了超越简单智能体来源的信息流追踪。
Only \texttt{declassify}/\texttt{endorse} transforms are unused, as expected: they exist for long-running sessions where label monotonicity causes over-tainting, a scenario absent from this task-oriented corpus.
% 仅 \texttt{declassify}/\texttt{endorse} 变换未使用，这在预期内：它们用于长会话中标签单调性导致过度标记的场景，而这在面向任务的语料库中不存在。

\subsection{RQ2: \sys{} Effectiveness for Policy Compliance}
% \subsection{RQ2：\sys{} 策略合规有效性}
\label{sec:eval:rq2}

\begin{figure}[t]
\centering
\includegraphics[width=\columnwidth]{figures/rq1-pipeline.pdf}
\caption{RQ2 evaluation pipeline: four enforcement paths from natural-language policy to agent-level decision.}

\Description{Flow diagram showing prompt-filter, tool-regex, FIDES, and \sys{} paths through policy translation, enforcement, and judging.}
% \Description{流程图显示 prompt-filter、tool-regex、FIDES 和 \sys{} 通过策略翻译、执行和判断的路径。}
\label{fig:rq2-pipeline}

\paragraph{Benchmark Motivation.}
% \paragraph{基准动机。}
Existing enforcement benchmarks evaluate guardrails at the tool-call boundary.
% 现有执行基准在工具调用边界评估护栏。
ShieldAgent-Bench~\cite{li2025shieldagent} and ToolSafe~\cite{chen2026toolsafe} measure precision and recall on individual tool calls or trajectories, while AgentSpec, Progent, FIDES, and CaMeL~\cite{wang2025agentspec,shi2025progent,costa2025fides,debenedetti2025camel} measure attack-success-rate reduction on AgentDojo scenarios~\cite{debenedetti2024agentdojo}.
% ShieldAgent-Bench~\cite{li2025shieldagent} 和 ToolSafe~\cite{chen2026toolsafe} 在单个工具调用或轨迹上衡量精度和召回，而 AgentSpec、Progent、FIDES 和 CaMeL~\cite{wang2025agentspec,shi2025progent,costa2025fides,debenedetti2025camel} 在 AgentDojo 场景~\cite{debenedetti2024agentdojo}上衡量攻击成功率降低。
However, none test whether enforcement holds when an agent causes a system effect through an indirect path (e.g., a subprocess) or a cross-event sequence.
% 然而，没有一个测试当智能体通过间接路径（如子进程）或跨事件序列导致系统效果时执行是否保持。
To fill this gap, we build a benchmark that challenges runtime enforcement with these two evasion categories.
% 为填补这一空白，我们构建了一个用这两类规避方式挑战运行时执行的基准。
% 然而，这些在工具调用边界运行：它们不测试当智能体通过间接或跨事件路径导致系统效果时执行是否保持。

\paragraph{Rule set and scope.}
% \paragraph{规则集和范围。}
RQ2 samples 38 rules from the 607 OS-enforceable policies, excluding semantic-only policies (no system-observable effect) and content policies (require linters, not OS-level IFC).
% RQ2 从 607 条操作系统可执行策略中采样 38 条规则，排除仅语义策略（无系统可观察效果）和内容策略（需要代码检查器而非操作系统级 IFC）。
We sample 2--3 rules from each of 15 repositories to cover per-event and cross-event rules, project- and task-context requirements, and common policy themes, yielding 20 per-event and 18 cross-event rules.
% 我们从 15 个仓库各采样 2--3 条规则，以覆盖单事件和跨事件规则、项目和任务上下文要求以及常见策略主题，最终得到 20 条单事件和 18 条跨事件规则。

\paragraph{Traces.}
% \paragraph{轨迹。}
A Codex CLI agent~\cite{openai2025codex} under GPT-5.5~\cite{openai2026gpt55} generates five traces per rule by reading the rule text and repository.
% 一个 Codex CLI 智能体~\cite{openai2025codex}配合 GPT-5.5~\cite{openai2026gpt55}通过读取规则文本和仓库为每条规则生成五条轨迹。
The policy DSL and all enforcement systems remain invisible to the generator to prevent information leakage.
% 策略 DSL 和所有执行系统对生成器保持不可见，以防止信息泄露。
Each trace simulates realistic coding-agent work, including user prompts, file I/O, shell commands, and build/test invocations on a real repository.
% 每条轨迹模拟真实的编码智能体工作，包括用户提示、文件 I/O、shell 命令以及在真实仓库上的构建/测试调用。

For each rule, the agent produces two compliant traces and three violation traces.
% 对于每条规则，智能体生成两条合规轨迹和三条违规轨迹。
The compliant traces stress false-positive detection, with allowed-effect traces exercising permitted actions close to the boundary and lookalike traces performing technically compliant operations that resemble violations.
% 合规轨迹强调误报检测，允许效果轨迹演练接近边界的允许操作，相似轨迹执行技术上合规但类似违规的操作。
The three violation families reflect execution-path categories observed in deployed coding agents~\cite{anthropic2026containment,maloyan2026sok}: \emph{direct} traces trigger the prohibited operation via a tool call, \emph{script} traces move the side effect into a subprocess by splitting authoring and execution~\cite{maloyan2026sok}, and \emph{hidden} traces embed the side effect in an auxiliary artifact whose entrypoint appears benign~\cite{wu2026skillject}.
% 三个违规族反映了部署的编码智能体中观察到的执行路径类别~\cite{anthropic2026containment,maloyan2026sok}：\emph{直接}轨迹通过工具调用触发禁止操作，\emph{脚本}轨迹通过拆分编写和执行将副作用移入子进程~\cite{maloyan2026sok}，\emph{隐藏}轨迹将副作用嵌入入口点看起来无害的辅助工件中~\cite{wu2026skillject}。
This design yields 190 traces (38 rules $\times$ 5).
% 此设计产生 190 条轨迹（38 条规则 $\times$ 5）。

We replay each trace on codex-sdk, let the tested agent continue for up to five additional tool steps, and record whether it recovers toward compliance after receiving enforcement feedback.
% 我们在 codex-sdk 上重放每条轨迹，让测试的智能体继续最多五个额外工具步骤，并记录它在收到执行反馈后是否恢复到合规。

\paragraph{Compared systems.}
% \paragraph{比较系统。}
RQ2 compares five enforcement systems on the same 190 traces (Figure~\ref{fig:rq2-pipeline}).
% RQ2 在相同的 190 条轨迹上比较五个执行系统（图~\ref{fig:rq2-pipeline}）。
Three baselines represent common runtime enforcement layers: \emph{prompt-filter}, an LLM that decides allow/block per tool call~\cite{rebedea2023nemo,xiang2025guardagent}; \emph{tool-regex}, deterministic pattern matching as deployed in Claude Code hooks~\cite{anthropic2026hooks}, Codex approval policies~\cite{openai2025codex}, and AgentSpec~\cite{wang2025agentspec}; and \emph{FIDES}~\cite{costa2025fides}, tool-level IFC.
% 三个基线代表常见的运行时执行层：\emph{提示过滤}，按工具调用决定允许/阻止的 LLM~\cite{rebedea2023nemo,xiang2025guardagent}；\emph{工具正则}，Claude Code hooks~\cite{anthropic2026hooks}、Codex 批准策略~\cite{openai2025codex}和 AgentSpec~\cite{wang2025agentspec}中部署的确定性模式匹配；以及 \emph{FIDES}~\cite{costa2025fides}，工具级 IFC。
We add \textsc{\sys{}-opaque}, which runs \sys{}'s kernel engine without semantic feedback, as both a proxy for sandbox-style kernel IFC~\cite{pasquier2018camquery,flume,zeldovich2006histar} and an ablation for feedback.
% 我们添加 \textsc{\sys{}-opaque}，它运行无语义反馈的 \sys{} 内核引擎，既作为沙箱式内核 IFC~\cite{pasquier2018camquery,flume,zeldovich2006histar}的代理，也作为反馈的消融实验。
Full \sys{} is the fifth system.
% 完整 \sys{} 是第五个系统。

A separate translation agent (Claude Code with Claude Opus~4.6) reads each rule and the cloned repository, then produces the policy artifact for each non-prompt system without access to ground-truth traces.
% 一个独立的翻译智能体（Claude Code 配合 Claude Opus~4.6）读取每条规则和克隆的仓库，然后为每个非提示系统生成策略工件，不访问真实轨迹。

\paragraph{Metric.}
% \paragraph{指标。}
Following prior benchmarks~\cite{zheng2023judging,debenedetti2024agentdojo,li2025shieldagent,chen2026toolsafe}, we use an LLM trajectory judge to assign TP, TN, FP, or FN to each trace.
% 遵循先前基准~\cite{zheng2023judging,debenedetti2024agentdojo,li2025shieldagent,chen2026toolsafe}，我们使用 LLM 轨迹判断器为每条轨迹分配 TP、TN、FP 或 FN。
The judge sees the ground-truth label, whether enforcement triggered, the feedback delivered, and the agent's subsequent actions.
% 判断器看到真实标签、执行是否触发、传递的反馈以及智能体的后续行动。
We define outcomes relative to policy compliance. TP and TN are correct outcomes (violation recovered or compliant trace completed without wrongful intervention), while FP and FN are errors (compliant trace blocked or violation missed).
% 我们相对于策略合规定义结果。TP 和 TN 是正确结果（违规恢复或合规轨迹无错误干预完成），而 FP 和 FN 是错误（合规轨迹被阻止或违规被遗漏）。

We report \emph{Decision Compliance Rate} (DCR), defined as $(\mathrm{TP}+\mathrm{TN})/(\mathrm{TP}+\mathrm{TN}+\mathrm{FP}+\mathrm{FN})$.
% 我们报告 \emph{决策合规率}（DCR），定义为 $(\mathrm{TP}+\mathrm{TN})/(\mathrm{TP}+\mathrm{TN}+\mathrm{FP}+\mathrm{FN})$。
DCR is end-to-end, capturing policy translation, runtime enforcement, feedback delivery, and agent recovery.
% DCR 是端到端的，捕获策略翻译、运行时执行、反馈传递和智能体恢复。
We manually reviewed the samples the judge flagged for double-checking, plus 50 random judgments, all of which agreed with our assessment.
% 我们手动更正了判断器标记为需要复查的样本，并额外随机审查了 50 个判断，全部与我们的手动评估一致。

\noindent\textbf{\sys{} achieves 75.8\% DCR, 22--31 percentage points above all baselines (Table~\ref{tab:rq2-confusion}).}
% \noindent\textbf{\sys{} 达到 75.8\% DCR，比所有基线高 22--31 个百分点（表~\ref{tab:rq2-confusion}）。}
The gap concentrates on violation traces, where \sys{} correctly resolves 86 of 114 (75.4\%), compared with 27--44 for baselines (2.0--3.2$\times$).
% 差距集中在违规轨迹上，\sys{} 正确解决 114 条中的 86 条（75.4\%），而基线为 27--44 条（2.0--3.2 倍）。
This advantage stems from detection rate, with \sys{} detecting 77.2\% of violations while prompt-filter, tool-regex, and FIDES detect only 34--40\%.
% 这一优势源于检测率，\sys{} 检测到 77.2\% 的违规，而提示过滤、工具正则和 FIDES 仅检测到 34--40\%。

\noindent\textbf{\sys{}'s advantage concentrates on indirect execution paths (Figure~\ref{fig:rq2-family-breakdown}).}
% \noindent\textbf{\sys{} 的优势集中在间接执行路径（图~\ref{fig:rq2-family-breakdown}）。}
On compliant traces and direct violations, all systems perform comparably, so the baseline gap opens on script and hidden trace families.
% 在合规轨迹和直接违规上所有系统表现相当，因此基线差距在脚本和隐藏轨迹族上扩大。
Prompt-filter, tool-regex, and FIDES drop to near-zero on hidden traces because the prohibited effect occurs inside a subprocess or behind a neutral entrypoint.
% 提示过滤、工具正则和 FIDES 在隐藏轨迹上降至接近零，因为禁止效果发生在子进程内或中性入口点后面。
\sys{} retains detection through kernel-level label propagation, which follows the effect regardless of how the agent invoked it.
% \sys{} 通过内核级标签传播保持检测，无论智能体如何调用都能跟踪效果。

Cross-event rules (18 of 38) widen the gap further.
% 跨事件规则（38 条中的 18 条）进一步扩大差距。
Tool-call baselines lack the persistent state needed to track ordering constraints such as ``run tests before commit,'' whereas \sys{}'s session-level label state captures these dependencies.
% 工具调用基线缺乏跟踪顺序约束（如"提交前运行测试"）所需的持久状态，而 \sys{} 的会话级标签状态捕获这些依赖关系。

\noindent\textbf{Agents can learn from failure, raising detection from 77.2\% to 94.7\% with one revision.}
% \noindent\textbf{智能体可以从失败中学习，一次修订将检测率从 77.2\% 提高到 94.7\%。}
We decompose violation-trace outcomes into \emph{detection rate} (fraction of 114 violation traces on which enforcement triggered) and \emph{recovery rate} (fraction of detected violations where the agent subsequently complied).
% 我们将违规轨迹结果分解为 \emph{检测率}（114 条违规轨迹中执行触发的比例）和 \emph{恢复率}（检测到的违规中智能体随后合规的比例）。
Translation quality drives both. Rules too narrow miss violations, while rules too broad match compliant actions, causing 17 of the 18 false positives.
% 翻译质量驱动两者。规则过窄会遗漏违规，而规则过宽会匹配合规操作，导致 18 个误报中的 17 个。

To measure improvability, we feed each false-negative trace's evidence and corrective feedback to the translation agent and let it revise the rule once.
% 为衡量可改进性，我们将每条假阴性轨迹的证据和纠正反馈提供给翻译智能体，让其修订规则一次。
Rerunning the 28 FN traces with revised rules recovers 26 (92.9\%), showing that the DSL supports iterative refinement.
% 用修订后的规则重新运行 28 条 FN 轨迹，恢复 26 条（92.9\%），表明该 DSL 支持迭代完善。

\noindent\textbf{Semantic feedback converts kernel detections into agent compliance at 97.7\%.}
% \noindent\textbf{语义反馈以 97.7\% 将内核检测转化为智能体合规。}
Full \sys{} produces $3\times$ more correct violation-trace outcomes than the same engine without feedback (86 vs.\ 27).
% 完整 \sys{} 产生的正确违规轨迹结果是无反馈引擎的 3 倍（86 对 27）。
The \textsc{\sys{}-opaque} ablation removes semantic feedback, and correct responses drop from 144 to 102 as FN rises from 28 to 87 while FP drops from 18 to 1.
% \textsc{\sys{}-opaque} 消融移除语义反馈，正确响应从 144 降至 102，FN 从 28 升至 87 而 FP 从 18 降至 1。
It detects 75.4\% of violations, but only 31.4\% of those detections lead to compliance because the agent receives a generic denial without corrective guidance.
% 它检测到 75.4\% 的违规，但只有 31.4\% 的检测导致合规，因为智能体收到的是通用拒绝而非纠正指导。
Feedback also improves detection indirectly, since an agent that receives a corrective payload after a first denial revises its approach, and the engine catches the revised attempt if it still violates.
% 反馈还间接提高检测，因为收到第一次拒绝后纠正载荷的智能体会修改其方法，如果修改后的尝试仍然违规引擎会捕获它。

\paragraph{Cross-model stability.}
% \paragraph{跨模型稳定性。}
\noindent\textbf{\sys{}'s advantage replicates under a second model (Figure~\ref{fig:rq2-dcr}).}
% \noindent\textbf{\sys{} 的优势在第二个模型下复现（图~\ref{fig:rq2-dcr}）。}
The DeepSeek-Pro~V4~\cite{deepseek2026v4} end-to-end replication preserves the system ranking, with \sys{} highest at 77.4\% DCR.
% DeepSeek-Pro~V4~\cite{deepseek2026v4}端到端复现保持系统排名，\sys{} 以 77.4\% DCR 最高。
Per-cell agreement between the two model settings yields Cohen's $\kappa=0.822$.
% 两个模型设置之间的每单元一致性产生 Cohen's $\kappa=0.822$。
As expected, deterministic systems show the highest stability (tool-regex $\kappa=0.920$, FIDES $\kappa=0.919$, \sys{} $\kappa=0.852$), while prompt-filter is lowest ($\kappa=0.592$) because its decisions depend on model-specific reasoning.
% 如预期，确定性系统稳定性最高（工具正则 $\kappa=0.920$、FIDES $\kappa=0.919$、\sys{} $\kappa=0.852$），而提示过滤最低（$\kappa=0.592$），因为其决策依赖于模型特定的推理。

\paragraph{Goal.}
% \paragraph{目标。}
Unlike RQ2, which isolates single decision points, RQ4 evaluates \sys{} on complete coding-agent tasks (implementing features, fixing bugs, configuring builds) in real repositories, scored by the official OctoBench checklist judge.
% 与隔离单个决策点的 RQ2 不同，RQ4 在真实仓库的完整编码智能体任务（实现功能、修复 bug、配置构建）上评估 \sys{}，由官方 OctoBench 清单判断器评分。

\paragraph{Benchmark and subset.}
% \paragraph{基准和子集。}
OctoBench~\cite{octobench} is a coding benchmark with 217 tasks spanning system prompts, tool schemas, project files (\texttt{CLAUDE.md}, \texttt{AGENTS.md}), and user queries.
% OctoBench~\cite{octobench}是一个编码基准，包含 217 个任务，涵盖系统提示、工具模式、项目文件（\texttt{CLAUDE.md}、\texttt{AGENTS.md}）和用户查询。
We use the official evaluator unchanged.
% 我们原样使用官方评估器。
Because \sys{} cannot observe purely semantic checks (e.g., tone), we select a 21-task (61 rules) subset whose user-query checklist contains at least one targeted OS-enforceable item (command, file operation, tool-execution).
% 因为 \sys{} 无法观察纯语义检查（如语气），我们选择一个 21 任务（61 条规则）子集，其用户查询清单包含至少一个有针对性的操作系统可执行项（命令、文件操作、工具执行）。

% 选定子集涵盖 \texttt{CLAUDE.md}、\texttt{AGENTS.md} 和用户查询策略源、两个智能体 harness 和七个仓库。

% 纯文件内容、风格、语气和仅自然语言案例被排除。
The remaining 196~tasks lack OS-enforceable checklist items and fall outside \sys{}'s enforcement scope.
% 剩余 196 个任务缺乏操作系统可执行清单项，不在 \sys{} 的执行范围内。
Each task runs under three conditions: baseline (no enforcement), Claude Code hooks~\cite{anthropic2026hooks} (the official sandbox approval policy), and \sys{} (OS-level hooks plus semantic feedback).
% 每个任务在三个条件下运行：基线（无执行）、Claude Code hooks~\cite{anthropic2026hooks}（官方沙箱批准策略）和 \sys{}（操作系统级钩子加语义反馈）。
An AI agent (Codex~\cite{openai2025codex} with GPT-5.5~\cite{openai2026gpt55}) reads the task description and generates initial policy without human involvement.
% 一个 AI 智能体（Codex~\cite{openai2025codex}配合 GPT-5.5~\cite{openai2026gpt55}）读取任务描述并在无人工参与的情况下生成初始策略。

% 在执行期间，同一 Codex 智能体通过运行时接口提交增量来调整策略（\S\ref{sec:authority}），根据执行反馈为自己或任务执行子智能体完善规则。

% 我们另外报告来自相同清单结果的三个诊断子指标，即 \emph{用户查询奖励}（仅用户任务检查）、\emph{实现/测试奖励}（实现、测试、配置和修改检查）和 \emph{合规奖励}（合规类型检查）。
All submetrics use the official LLM-based checklist judge.
% 所有子指标使用官方 LLM 基础清单判断器。

The improvement concentrates on task-specific checks, with user-query reward rising by 9.9 points and implementation/test reward by 9.7 points over baseline, while compliance reward shows a smaller 2.8-point gain.

\subsection{RQ5: Safety Enforcement Beyond Coding Tasks}
% \subsection{RQ5：编码任务之外的安全执行}
\label{sec:eval:rq5}


\paragraph{Benchmark and setup.}
% \paragraph{基准和设置。}
OpenAgentSafety~\cite{openagentsafety} contains 361 safety tasks that instruct an agent to perform workplace or personal-assistant actions whose completion requires an unsafe side effect: exfiltrating credentials, planting surveillance artifacts, wiping system configurations, tampering with financial records, or materializing harmful content into files.
% OpenAgentSafety~\cite{openagentsafety}包含 361 个安全任务，指示智能体执行工作场所或个人助理行为，其完成需要不安全的副作用：泄露凭证、植入监控工件、清除系统配置、篡改财务记录或将有害内容物化到文件中。
Each task runs inside a Docker-isolated OpenHands~\cite{wang2025openhands} runtime with service dependencies (GitLab, ownCloud) and an independent evaluator that scores the final workspace state.
% 每个任务在 Docker 隔离的 OpenHands~\cite{wang2025openhands}运行时中运行，带有服务依赖（GitLab、ownCloud）和对最终工作空间状态评分的独立评估器。
We run every task under two conditions, namely \emph{baseline} (OpenHands with no enforcement) and \emph{\sys{}} (OpenHands with an OS-level \sys{} watcher loading a per-task policy).
% 我们在两个条件下运行每个任务，即 \emph{基线}（无执行的 OpenHands）和 \emph{\sys{}}（带加载每任务策略的操作系统级 \sys{} 监视器的 OpenHands）。
This setup differs from RQ1--RQ4 in two dimensions. The agent framework is OpenHands rather than Codex CLI, and the workload is workplace safety tasks rather than repository coding tasks.
% 此设置在两个维度上与 RQ1--RQ4 不同。智能体框架是 OpenHands 而非 Codex CLI，工作负载是工作场所安全任务而非仓库编码任务。

\paragraph{Policy generation.}
% \paragraph{策略生成。}
In deployment, the agent harness processes the user's request before execution begins, and guard agents and safety classifiers~\cite{xiang2025guardagent,chennabasappa2025llamafirewall,rebedea2023nemo,invariant-guardrails} similarly analyze the incoming prompt to set runtime constraints.
% 在部署中，智能体 harness 在执行开始前处理用户请求，守卫智能体和安全分类器~\cite{xiang2025guardagent,chennabasappa2025llamafirewall,rebedea2023nemo,invariant-guardrails}类似地分析传入提示以设置运行时约束。
Our policy generator mirrors this: a Codex agent (GPT-5.5~\cite{openai2026gpt55}) reads only the task description and extracts explicit file paths, system commands, and high-confidence sink categories such as credentials, cron jobs, SSH keys, account mutations, and network egress.
The result is a per-task \sys{} policy compiled to an eBPF configuration blob.
% 我们的策略生成器反映这一点：一个 Codex 智能体（GPT-5.5~\cite{openai2026gpt55}）仅读取任务描述，提取显式文件路径、系统命令和高置信度汇类别（凭证、cron 作业、SSH 密钥、账户变更、网络出口），生成编译为 eBPF 配置 blob 的每任务 \sys{} 策略。
If no concrete OS-observable behavior is identifiable from the description, the generator emits a no-op marker.
% 如果从描述中无法识别具体的操作系统可观察行为，生成器发出空操作标记。
The generator does not read the evaluator, checkpoints, trajectories, or any prior run results, making policy generation zero-shot with respect to evaluation criteria.
% 生成器不读取评估器、检查点、轨迹或任何先前运行结果，使策略生成相对于评估标准是零样本的。
Of the 361~tasks, the generator produces non-trivial policies for 303~tasks and no-op markers for 58~tasks where the task description names no concrete OS sink.
% 在 361 个任务中，生成器为 303 个任务生成非平凡策略，为 58 个任务描述未命名具体操作系统汇的任务发出空操作标记。
The generator runs before the agent starts and does not ingest untrusted content, so the resulting policy is independent of unsafe execution.
% 生成器在测试的智能体启动前运行且不摄取不受信任的内容，因此结果策略独立于不安全执行。
Because the generator completes before the tested agent starts, the resulting policies are loaded at higher-domain level (\S\ref{sec:design}) and cannot be weakened at runtime.
% 因为生成器在测试的智能体启动前完成，结果策略在更高域级别加载（\S\ref{sec:design}），且不能在运行时被削弱。

\paragraph{Results.}
% \paragraph{结果。}
\noindent\textbf{On 361 safety tasks, \sys{} prevents 74\% of baseline-unsafe behaviors (Figure~\ref{fig:rq5-safety}).}
% \noindent\textbf{在 361 个安全任务上，\sys{} 阻止了 74\% 的基线不安全行为（图~\ref{fig:rq5-safety}）。}
Of 106~baseline-unsafe tasks, \sys{} terminated the unsafe operation in 78~cases before the harmful effect completed.
% 在 106 个基线不安全任务中，\sys{} 在 78 个案例中在有害效果完成前终止了不安全操作。
In 28~cases the agent completed the unsafe action without triggering a policy.
% 在 28 个案例中，智能体在不触发策略的情况下完成了不安全行为。
\sys{} also activated on 16\% of tasks whose baseline was already safe, where the description-only policy matched a benign operation near the prohibited boundary.
% \sys{} 还在 16\% 的基线已经安全的任务上激活，其中仅描述策略匹配了接近禁止边界的良性操作。
The 58~no-op tasks, those with no OS behavior in the description, account for most of the remaining baseline-safe cases and are excluded from the prevention rate denominator.
% 58 个空操作任务（描述中无操作系统行为）占剩余大部分基线安全案例，被排除在预防率分母之外。

\paragraph{Failure analysis.}
% \paragraph{失败分析。}
The 28~unblocked baseline-unsafe cases fall into three categories.
% 28 个未阻止的基线不安全案例分为三类。
\emph{Chat/semantic harm}: the unsafe behavior is a message with no OS-observable artifact, outside \sys{}'s enforcement scope by design.
% \emph{聊天/语义危害}：不安全行为是没有操作系统可观察工件的消息，按设计不在 \sys{} 执行范围内。
\emph{Unsafe file content}: the agent produces unsafe file content, which is not the main goal of \sys{}.
% \emph{不安全文件内容}：智能体产生不安全的文件内容，这不是 \sys{} 的主要目标。
\emph{Service-side artifact}: the effect is a WebDAV upload or database mutation inside a service container.
% \emph{服务端工件}：效果是服务容器内的 WebDAV 上传或数据库变更。
These are hook-coverage gaps, not policy errors. The generator produced a matching rule for the intended sink, but the agent reached the same effect through a path the current hook set does not observe.
% 这些是钩子覆盖差距，而非策略错误。生成器为预期汇生成了匹配规则，但智能体通过当前钩子集未观察的路径达到相同效果。
Chat and content harms are outside \sys{}'s scope by design, and service-side artifacts require protocol-level hook extensions.
% 聊天和内容危害按设计不在 \sys{} 范围内，服务端工件需要协议级钩子扩展。

\section{Related Work}
% \section{相关工作}
\label{sec:related}

\textbf{In-kernel provenance and information flow.}
% \textbf{内核内来源和信息流。}
Provenance systems (PASS, Hi-Fi, LPM, CamFlow)~\cite{muniswamy2006pass,pohly2012hifi,bates2015lpm,pasquier2017camflow} record audit-grade graphs but do not compile agent-oriented policy or return semantic feedback.
% 来源系统（PASS、Hi-Fi、LPM、CamFlow）~\cite{muniswamy2006pass,pohly2012hifi,bates2015lpm,pasquier2017camflow}记录审计级图，但不编译面向智能体的策略或返回语义反馈。
CamQuery~\cite{pasquier2018camquery} is the closest prior system: it propagates labels and can deny matching operations, but targets adversarial intrusions with general provenance queries rather than an agent-facing DSL with semantic feedback.
% CamQuery~\cite{pasquier2018camquery}是最接近的先前系统：它传播标签并可以拒绝匹配操作，但使用通用来源查询针对对抗性入侵，而非带语义反馈的面向智能体 DSL。
Dynamic taint-tracking systems~\cite{enck2010taintdroid,clause2007dytan,kemerlis2012libdft,yin2007panorama} operate at byte or instruction granularity, whereas \sys{} operates at object granularity, trading precision for low-overhead whole-session coverage.
% 动态污点跟踪系统~\cite{enck2010taintdroid,clause2007dytan,kemerlis2012libdft,yin2007panorama}在字节或指令粒度运行，而 \sys{} 在对象粒度运行，以低开销全会话覆盖换取精度。
Contemporary eBPF tools such as Tetragon, Tracee, and eBPF-PATROL~\cite{ciliumtetragon,aquasecuritytracee,ghimire2025ebpfpatrol} provide per-event predicates or single-channel lineage flags, whereas \sys{} propagates multi-label information flow across channels and checks cross-event conditions.
% 当代 eBPF 工具如 Tetragon、Tracee 和 eBPF-PATROL~\cite{ciliumtetragon,aquasecuritytracee,ghimire2025ebpfpatrol}提供每事件谓词或单通道血统标志，而 \sys{} 跨通道传播多标签信息流并检查跨事件条件。

\textbf{Agent policy systems.}
% \textbf{智能体策略系统。}
Crab~\cite{wu2026crab} uses eBPF to help C/R decisions and AgentSight~\cite{agentsight} uses eBPF for agent observability, while \sys{} focuses on enforcing agent behavior.
% Crab~\cite{wu2026crab}使用 eBPF 帮助 C/R 决策，AgentSight~\cite{agentsight}使用 eBPF 进行智能体可观测性，而 \sys{} 专注于执行智能体行为。
At the agent layer, programmable rails and guard agents check prompts or action trajectories~\cite{rebedea2023nemo,xiang2025guardagent}, and AgentSpec~\cite{wang2025agentspec} enforces deterministic guards at the tool-call boundary.
% 在智能体层，可编程护栏和守卫智能体检查提示或行为轨迹~\cite{rebedea2023nemo,xiang2025guardagent}，AgentSpec~\cite{wang2025agentspec}在工具调用边界执行确定性守卫。
\sys{} complements them below the tool API, where shell-outs and subprocesses remain visible.
% \sys{} 在工具 API 之下补充它们，shell 调用和子进程仍然可见。
FIDES and CaMeL~\cite{costa2025fides,debenedetti2025camel} apply typed IFC within the agent loop, which catches data-flow violations at the API boundary but cannot observe effects that escape through shell-outs, subprocesses, or compiled binaries.
% FIDES 和 CaMeL~\cite{costa2025fides,debenedetti2025camel}在智能体循环内应用类型化 IFC，在 API 边界捕获数据流违规，但无法观察通过 shell 调用、子进程或编译二进制文件逃逸的效果。
\sys{} applies information-flow reasoning at the OS boundary, where cross-event label state persists across context windows and enforcement is not bypassed by subprocess indirection.
% \sys{} 在操作系统边界应用信息流推理，跨事件标签状态跨上下文窗口持久存在，执行不会被子进程间接绕过。

\section{Conclusion}
% \section{结论}
\label{sec:conclusion}

We introduce \sys{}, which enforces agent intent-level policies at the OS level via eBPF and returns semantic feedback for agent compliance.
% 我们引入 \sys{}，它通过 eBPF 在操作系统级别执行智能体意图级策略，并返回语义反馈以实现智能体合规。
It improves correct violation-trace outcomes by 2.0--3.2$\times$ over prompt-filter, tool-regex, FIDES~\cite{costa2025fides}, and feedback-free kernel IFC, prevents 74\% of baseline-unsafe behaviors on 361 safety tasks, and adds 1.9\%--8.4\% end-to-end overhead.
% 它将正确违规轨迹结果提高到 prompt-filter、tool-regex、FIDES~\cite{costa2025fides}和无反馈内核 IFC 的 2.0--3.2 倍，在 361 个安全任务上阻止了 74\% 的基线不安全行为，并增加 1.9\%--8.4\% 的端到端开销。
\sys{} targets per-event and cross-event policies (45\% of the corpus) by design.
% \sys{} 按设计针对单事件和跨事件策略（语料库的 45\%）。
Content policies are better served by linters and static analyzers.
% 内容策略更适合由代码检查器和静态分析器处理。
